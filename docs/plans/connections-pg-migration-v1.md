# Workstream B — connections (MP-declaration graph) → Postgres

**Status:** design (2026-07-04). Foundation for the last zero-PG domain. Follows the
proven procurement pattern (schema → loader → SQL API → `/api/db` → migrate hooks →
retire JSON). See [direct-db-ingest-v1.md](direct-db-ingest-v1.md) for the umbrella
plan and the AI-second-consumer rule.

## 1. What this domain is (and isn't)

The **MP-declaration business-interest graph**: companies an MP/official owns or
manages, from Сметна палата asset declarations (`stakes`) + Commerce-Registry
filings (`mpRoles` / TR roles). Sourced under `data/parliament/connections*` +
`data/parliament/companies-index.json` + `data/officials/derived/company_links.json`,
generated by `scripts/declarations/build_connections_graph.ts` +
`build_officials_connections.ts`.

**Not** the same as the procurement `company_politicians` table already in PG — that
is the **curated high-confidence subset** the connections pipeline emits *for
procurement cross-referencing*. This workstream models the **fuller graph** (all
stakes + all TR roles + medium confidence + HQ resolution + officials). Once it
lands, `company_politicians` could even be derived from it, but that's out of scope
here (keep it as-is to avoid disturbing the shipped procurement tools).

## 2. Core data model (from `CompanyIndexEntry`)

```
company (slug) ──< edge >── politician (mp_id | official slug)
```
- **company**: `slug` (canonical), `display_name`, `registered_offices[]`,
  `ekatte_hq[]` (resolved HQ settlement, Sofia → 68134), `obshtina_hq`,
  `hq_match_quality` (high|medium|foreign|unresolved), `eik` (when TR-matched).
- **edge (stake)**: mp_id, declarant_name, declaration_year, fiscal_year,
  institution, source_url, stake detail (share %, value).
- **edge (TR role)**: mp_id/official, role (`manager|partner|tr_owner|procurator|…`),
  is_current, confidence (high|medium).
- **official edges**: same shape keyed by official `slug` (from `company_links.json`).

## 3. Proposed schema (`schema/pg/041_connections.sql`)

```sql
CREATE TABLE decl_companies (
  slug            text PRIMARY KEY,
  display_name    text NOT NULL,
  name_fold       text NOT NULL,            -- translit_bg_latin(display_name) for trgm
  eik             text,                     -- when TR-matched (nullable)
  registered_office text,
  ekatte_hq       text,                     -- primary resolved HQ settlement
  obshtina_hq     text,
  hq_match_quality text
);
CREATE TABLE decl_edges (
  slug            text NOT NULL REFERENCES decl_companies(slug),
  ref             text NOT NULL,            -- '/candidate/mp-<id>' | '/officials/<slug>'
  kind            text NOT NULL,            -- 'mp' | 'official'
  politician_name text NOT NULL,
  party           text,                     -- current party (mp) for the party matrix
  source          text NOT NULL,            -- 'stake' | 'tr'
  role            text,                     -- TR role, or 'stake'
  is_current      boolean,
  confidence      text,                     -- 'high' | 'medium'
  declaration_year int,
  institution     text,
  source_url      text,
  stake           jsonb                     -- raw stake detail when source='stake'
);
-- Index BOTH sides of every join (reference/pg_query_performance):
CREATE INDEX ON decl_edges (slug);
CREATE INDEX ON decl_edges (ref);
CREATE INDEX ON decl_edges (kind);
CREATE INDEX ON decl_companies (ekatte_hq);
CREATE INDEX ON decl_companies (obshtina_hq);
CREATE INDEX ON decl_companies USING gin (name_fold gin_trgm_ops);
CREATE INDEX ON decl_companies (eik);
```

Determinism: follow [pg-payload-determinism](pg-query-performance.md) — ROUND sums,
rounded sort keys + slug/ref tiebreaks, `COLLATE "C"` on MIN(name).

## 4. Loader (`scripts/db/load_connections_pg.ts`)

Full rebuild, exactly like `load_pg`/`load_tr_pg`: truncate → COPY from
`companies-index.json` (mp edges) + `officials/derived/company_links.json` (official
edges). `party` comes from the parliament roster join (mp_id → current party) that
the party-matrix build already does. Add to `db:refresh`. `companies-index.json` +
`company_links.json` become **load sources that stay JSON** (Workstream C) — do NOT
retire them.

## 5. SQL API + `/api/db` routes (one per current hook)

| Hook (JSON today) | PG function | Route |

| `useConnectionsStats` | `connections_stats()` | `connections-stats` |

**AI second consumer (per §1b):** `ai/tools/*` read `connections-rankings.json` +
`-top.json`. Migrate those via the existing `fetchDb` seam once the routes exist
(same pattern as the 9 procurement tools) — verify against the harness.

## 6. Sequence (each step verified before the next)

1. **Schema + loader** — write `041`, `load_connections_pg.ts`; load; verify row
   counts + a sample company's edges vs `companies-index.json` (parity script, like
   the procurement migrations).
2. **SQL API** — one function at a time, each parity-checked against its JSON payload
   (rankings → `connections-rankings.json`, etc.).
3. **Routes** — add to `db_routes.js` (+ a `decl_companies` REGISTRY entry).
4. **Migrate hooks** — repoint the 11 hooks one screen at a time behind the data-hook
   seam; the force-directed `connections.json` graph may stay JSON.
5. **Migrate the AI connections tools** to `fetchDb` (harness-verified).
6. **Retire JSON** — once every reader (src + ai) of a connections file is on PG,
   git-untrack it (Workstream-A treatment). KEEP `companies-index.json` +
   `company_links.json` (load sources).

## 7. Effort + risk

Large but low-risk-per-step — it reuses the exact procurement machinery (`fetchDb`
seam, the parity-check recipe, the DbDataTable registry, the pg_trgm search, the
untrack teardown). The one genuinely new modelling decision is the **stake vs TR-role
edge unification** (§2/§3) — a single `decl_edges` table with a `source` discriminator
handles both. Officials reuse the same table via `kind='official'`.

**Prod:** `db:push` (new tables + functions) + `functions:db` redeploy, same as every
prior step.
