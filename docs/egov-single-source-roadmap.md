# data.egov.bg като единен източник — обратна връзка към МИДТ

**Роля на подателя:** Наясно (naiasno.bg / electionsbg.com) поддържа автоматизиран
ежедневен наблюдател над ~90 официални източника на данни за България и ги
поглъща в единна база. Този документ обобщава какво научихме за data.egov.bg от
практическото им използване в продукция, и предлага пътна карта за превръщането на
портала в реален единен източник (single source) на публични данни.

> Бележка за адресата: порталът data.egov.bg се оперира от бившата ДАЕУ, вече
> влята в МИДТ — така че МИДТ е точният адресат за платформените поправки. Част
> от препоръките по съдържанието изискват и координация с публикуващите ведомства
> (МФ, АОП, НЗОК, НОИ, НСИ, Сметна палата и др.).

Обхват на нашия наблюдател: **90 източника**, от които днес само **9** идват през
data.egov.bg. Останалите 81 се събират чрез скрейпване, PDF/XLS парсване или
заобикаляне на защити — точно тези 81 очертават потенциала за централизация.

---

## 0. Обобщение (TL;DR)

- **Платформата има системни технически дефекти**, които правят автоматизирания
  достъп ненадежден: счупен per-resource download, счупени документирани API
  маршрути, „тихо" връщане на HTML вместо данни с код 200, гео-блокиране на
  не-български IP адреси. Тези дефекти струват на всеки консуматор
  workaround-и — вж. Раздел 1.
- **9 датасета вече са на портала, но 5 от тях са непълни или счупени по
  съдържание** (АОП, Търговски регистър, ДФЗ, общинско изпълнение) — вж. Раздел 1.
- **Огромен корпус публични данни съществува, но е разпръснат** по сайтове на
  ведомства в PDF/XLS вид и може да се централизира с умерено усилие — Раздел 2.
- **Няколко ключови регистъра са заключени** зад Cloudflare/CAPTCHA/анти-скрейпинг
  въпреки че по закон са публични — Раздел 3.
- **Няколко високостойностни набора изобщо не съществуват** в машинно-четим вид —
  Раздел 4.

## 1. Данни, които ВЕЧЕ са на data.egov.bg, но са счупени / недостатъчни / непълни

### 1A. Платформени дефекти (засягат ВСИЧКИ датасети)

| # | Дефект | Наблюдение (дата на потвърждение) | Препоръка |
|---|--------|-----------------------------------|-----------|
| 1 | **Per-resource download е счупен** | `GET /resource/download/{uuid}/{fmt}` и POST формата 302-пренасочват към HTML shell с flash „Грешка при вземане на метаданни за ресурс" за портал-хоствани ресурси (потвърдено 2026-06-28 от български IP — не е гео-проблем). Състоянието е **интермитентно** — един и същ ресурс сервира ту данни, ту HTML. | Възстановяване на per-resource file backend с приоритет. Това е най-скъпият дефект за всеки консуматор. |
| 2 | **Документираните CKAN API маршрути не работят** | GET `searchDataset`, `getOrganisations` връщат `{success:false, "Непознат метод"}`. Работят само недокументираните POST `listDatasets` / `getResourceData` / `listOrganisations`. | Документиране на работещите endpoint-и; поправка или премахване на счупените, за да не подвеждат. |
| 3 | **Тихо отравяне на кешове (HTTP 200 + HTML)** | По време на прекъсване порталът връща **HTTP 200 с тялото на началната страница** вместо CSV. Консуматори, които валидират само `res.ok`/размер, записват HTML като „.csv" (потвърдено: `raw_data/indicators/mon/*.csv` = home page, 174 KB, разлика само в CSRF токена). | Връщане на коректни HTTP кодове (404/503) при липса на ресурс — никога 200+HTML. |
| 4 | **SPA дълбоки връзки се пренасочват** | Дълбоки връзки към dataset/resource страници препращат към SPA shell; няма стабилен permalink към конкретен ресурс — затова всички парсваме listing HTML. | Стабилни, машинно-достъпни permalink-ове (или чист JSON на dataset URL). |
| 5 | **Гео-блокиране (403) на не-BG IP** | Bare Apache 403 дори за началната страница при egress извън България (потвърдено от EU/Starlink IP; EU IP също е блокиран). Пречи на mirror-и, CI, външни реюзъри. | Премахване или документиране на гео-правилото; поне позволяване на EU IP диапазони. |
| 6 | **Bulk-zip е единствената надеждна пътека** | Работи `GET /dataset/{uuid}/resources/download/{csv|json}` → zip, но само на ниво цял датасет — няма per-resource zip (всички 404). За инкрементални дневни файлове (ТР) това налага теглене на целия ~540 MB архив. | Per-resource zip endpoint за инкрементален достъп. |

### 1B. Датасети на портала с проблеми по съдържанието

| Датасет (орг) | Проблем | Препоръка към публикуващото ведомство |
|---------------|---------|----------------------------------------|
| **АОП — обществени поръчки** (org 502) | Годишните CSV са непълни: 2018 е бил **0 договора** (наложи скрейпване на стария РОП web register); 2020/2021 са фрагментарни; ~34 000 синтетични „-x" дубликати; колизии на договорни ключове; **заглавия на договори и обособени позиции са отрязани в източника**. OCDS фортнайт-бъндъли и годишни CSV имат различна структура. | Пълни исторически серии (вкл. pre-2020 РОП); премахване на legacy дубли; неотрязани заглавия; единна схема OCDS за всички години. |
| **Търговски регистър** (дневни filings) | Bulk zip **спира на 2022-09-02**; данни за длъжностни лица/органи липсват за **~63%** от фирмите; съветите на ЮЛНЦ (НПО) не са парснати в структуриран вид. | Пълен исторически bulk; структуриран officer/board feed; парсване на 12d/13g/15b секциите за НПО. |
| **ДФ „Земеделие" — субсидии** (org 56) | Само финансови години **2015–2023**. 2024/2025 съществуват само в интерактивния регистър СЕУ (seu.dfz.bg) и **там без ЕИК колона** — къса връзката към субекта. | Публикуване на текущите години на портала с ЕИК ключ. |
| **Общинско касово изпълнение (ЕБК)** | Само **шепа общини** публикуват (Русе, Николаево); останалите 263 липсват. | Задължителен единен B3 ЕБК feed за всички общини. |
| **МОН — ДЗИ / НВО** | На портала, но като per-session CSV с дрейф на формата; уязвими на дефект №3 по-горе. | Стабилна схема + валидиран брой редове на година. |
| **ИАОС — качество на въздуха** (org 76) | Тримесечните CSV с ФПЧ10/ФПЧ2.5 по станции **не носят координати** — станциите се идентифицират само по име във формат „община - име на станция" (напр. „София - Хиподрума"), което налага парсване на името и справка в отделен списък на общините за геолокация (потвърдено при изграждането на националната карта на станциите за /sector/environment, 2026-07-16); ~14 фонови/планински станции изобщо нямат община. Публикуват се само ФПЧ10/ФПЧ2.5, макар порталът да има и NO₂/O₃/SO₂/CO датасети. | Добавяне на колони lat/lon (или EKATTE/община-код) към записа на станция; публикуване на всички замърсители на еднаква честота с обща схема. |

## 2. Данни, които НЕ са на data.egov.bg, но съществуват публично и могат да се централизират

Всички изброени се събират днес чрез скрейпване/PDF парсване от сайта на съответното
ведомство. Общата поправка: **публикуване като машинно-четим CSV/JSON със стабилна
схема, `updated_at` и канонични ключове (ЕИК / ЕКАТТЕ / BULSTAT) на портала.**

| Домейн | Днешен източник | Какво да се централизира | Специфична поправка |
|--------|-----------------|---------------------------|----------------------|
| **Тръжна фаза на поръчки (ЦАИС ЕОП)** | storage.eop.bg (плоски файлове) | Корпус процедури/обявления + статистика на офертите | Публикуване като датасет, не като raw storage bucket |
| **Партийно финансиране** (Сметна палата) | bulnao HTML/PDF | Годишни отчети на партии; статус на подаване | Структуриран JSON вместо HTML таблици |
| **Кампанийно финансиране (ЕРИК)** | erik.bulnao HTML | Дарители, дарения към кандидати, отчети по избори | Единен датасет per избори |
| **НЗОК / НЗОК плащания** | nhif.bg PDF/XLS | Плащания по болница (БМП), реимбурсиране по INN, месечен B1, единични цени на лекарства, клинична активност | Висок приоритет — health spending е ключова прозрачност |
| **НОИ / КФН пенсии** | nssi.bg / fsc.bg (XLSX-в-ZIP) | Изпълнение по фондове (B1), пенсионен годишник, частни фондове (стълб 2/3) | CSV на портала вместо XLSX-в-ZIP |
| **Социално подпомагане (АСП)** | asp.government.bg — годишен „Отчет за дейността" (**само PDF**; раздел „Отчети и доклади") | Помощи по вид (детски надбавки ЗСПД, помощи за хора с увреждания ЗХУ, целева помощ за отопление, ГМД/месечни соц. помощи, еднократни) — получатели/случаи + изплатена сума | **Потвърдено 2026-07-17:** АСП е публикувач на портала, но качва **само регистри** (напр. Регистър на доставчиците на социални услуги, ресурс `2a97b078-da71-4c3f-a307-1ba5b4fc952e`) — **нула датасети със статистика на помощите** (потвърдено с обхождане по ключови думи на целия портал). Данните са само национални/годишни в PDF отчетите. Числата **са машинно-извлекаеми** (`pdftotext` + regex — извлякохме получатели/суми за отопление, детски, ГМД и увреждания и ги валидирахме срещу отчета за 2025 г.), тоест агенцията вече разполага със структурираните числа — публикуването им като CSV е тривиално. **Per-област разбивка съществува вътрешно** (147 дирекции „Социално подпомагане" през 28 РДСП, тримесечни анализи по области), но **не се публикува** — единственият път е ЗДОИ. Публикуване като структуриран датасет per-помощ × година × **област** (получатели + сума, ЕКАТТЕ ключ) — това е ~37% от държавния разход (соц. защита), а разбивката по помощи и области днес не се вижда никъде. |
| **ИСУН 2020 — регистър на проектите** | 2020.eufunds.bg (XLSX експорт) | Договорите по еврофондове: бенефициент, програма, суми и **място на изпълнение** | **Потвърдено 2026-07-17** (при изграждането на картата по области за /sector/regional): полето „Местонахождение" е **свободен текст без ЕКАТТЕ ключ** — налага разрешаване по име срещу списъка на населените места (разбиване по запетаи с отчитане на скобите, парсване на подсказките „(общ.X, обл.Y)", alias карта за случаите София/Добрич, tiebreaker по област). По-същественото: от 81 693 договора **3 778 (4,6%) са с национален или регионален обхват и не могат да се отнесат към община — но носят €22,1 млрд. от €43,9 млрд., т.е. ~50% от корпуса**. Тоест половината европейски пари са нерегионализируеми по конструкция и всяка регионална карта от ИСУН показва само другата половина. Публикуване с ЕКАТТЕ ключ на мястото на изпълнение + експлицитен флаг за обхват (национален/регионален/местен), а за националните програми — разпределение по области, където съществува. |
| **Съдебна статистика (ВСС)** | ВСС годишни PDF таблици | Натовареност/движение на делата по съд | Табличен източник вместо PDF |
| **Административен регистър (ИИСДА)** | iisda.government.bg | Регистър на адм. услуги, годишен доклад за администрацията, органи на управление | Иронично — това е самият регистър на услугите, а няма open-data feed |
| **НСИ серии** | nsi.bg + частичен JSON-stat | Население, естествено движение, земеползване, регионални | Infostat е Cloudflare-walled (вж. Раздел 3); пренасяне на серии към портала |
| **Заетост / безработица (АЗ)** | az.government.bg stats | Регистрирана безработица по община | CSV feed |
| **Цени (КЗП „Колко струва")** | kolkostruva.bg opendata ZIP | Дневни малотърговски цени (има opendata, но не на портала) | Огледало на портала |
| **Култура (НФЦ/НФК)** | nfc.bg / ncf.bg XLS/новини | Филмови субсидии, грантове, състави на комисии | Структуриран регистър |

Публикуване като структуриран датасет per-проект, с ЕИК/община на бенефициента — това са публични средства извън обществените поръчки, които днес не се виждат никъде агрегирано. |

## 3. Данни, които съществуват в институция, но са ЗАКЛЮЧЕНИ / зад стена

Публични по закон, но недостъпни за автоматизирано събиране — изискват заобикаляне
на защити (Cloudflare/CAPTCHA/анти-скрейпинг) или са в нечетим PDF.

| Регистър | Стена | Защо е проблем |
|----------|-------|----------------|
| **Декларации Сметна палата** (register.cacbg.bg) | Per-record, анти-скрейпинг, без bulk | Имуществени/интереси декларации на публични лица — публични, но недостъпни масово |
| **КЗК жалби по ЗОП** (reg.cpc.bg) | Cloudflare + pager, изисква headed браузър | Обжалвания на поръчки |
| **Резултати ЦИК** (results.cik.bg) | Cloudflare Turnstile | Изборни резултати — налага Playwright bypass |
| **ГФО документи (ТР)** (portal.registryagency.bg) | Per-ActID PDF, частично CAPTCHA | Годишни финансови отчети на фирми |
| **Акцизен регистър (Митници)** (BACIS) | REST зад институционална стена | Лицензирани акцизни субекти |
| **Поименни гласувания общински съвети** | protokol PDF **без ToUnicode** (нечетим текст) | Технически неизвлекаем текст — дори OCR е нужен |
| **Декларации ИВСС** (магистрати) | Само page-scrape | Имуществени декларации на магистрати |
| **НСИ Infostat** | Cloudflare-walled зад отворения портал | Пълните регионални серии |
| **data.egov.bg сам по себе си** | Гео-403 за не-BG IP (вж. 1A-№5) | Порталът стена собствените си данни за външни реюзъри |

Препоръка: за регистри, които по закон са публични (декларации, изборни резултати,
ГФО), стените (Cloudflare/CAPTCHA) са в противоречие с целта — да се публикуват като
bulk open data на портала, вместо да се защитават срещу масов достъп.

## 4. Данни, които НЕ съществуват никъде, но биха били полезни

Няма ги в машинно-четим вид никъде — това е дневният ред за нови open-data продукти.

| Набор | Защо липсва / защо е ценен |
|-------|-----------------------------|
| **Данни на ниво оферта в поръчките** | Днес е само брой оферти; кой е участвал и с каква цена не се публикува дори от АОП. Ключово за конкуренция/картели. |
| **Консолидиран граф собственост↔лице↔политик** | Свързване ЕИК ↔ физическо лице ↔ публична длъжност през ТР + декларации + поръчки. Днес всеки прави свой crosswalk. |
| **ИСУН проекти: изпълнители + датирани плащания** | Публикуват се само суми на бенефициент, не изпълнители и график на плащанията. |
| **Поименни общински гласувания (структурирано)** | РМС 436 освобождава; никъде не съществува структурирано. |
| **Изпълнение на програмен бюджет (структурирано, admin+program)** | Само наративни PDF по министерства. |
| **Ефективност на пътни разходи (€/km)** | Няма връзка АПИ договори ↔ реално построени километри. |
| **Единични цени на лекарства по болница** | Само частично реконструирано; няма авторитетен feed. |
| **Финансови показатели на болниците (ЕЕОФ)** | Тримесечни, но нестандартизирани; case-mix знаменател липсва. |
| **Регистър на сделките с имоти / входове за AVM** | Агенцията по вписвания продава, не отваря — няма пазарни данни. |
| **Връзка възлагане→плащане** | Присъдена поръчка vs реално изплатено — не се проследява. |
| **Каноничен crosswalk ЕИК/ЕКАТТЕ/BULSTAT** | Няма официална таблица за свързване между регистрите — всеки реюзър я строи наново. |

## 5. Предложена пътна карта за МИДТ

**Фаза 1 — Поправка на платформата (най-висок ROI, засяга всичко):**
1. Възстановяване на per-resource download (1A-№1).
2. Коректни HTTP кодове вместо 200+HTML (1A-№3).
3. Документиране на работещите POST API + поправка/премахване на счупените GET (1A-№2).
4. Премахване/документиране на гео-403 (1A-№5).

**Фаза 2 — Довършване на съществуващите датасети (1B):**
Пълни исторически серии за АОП; пълен ТР bulk + officers; текущи ДФЗ години с ЕИК;
общинско изпълнение за всички общини.

**Фаза 3 — Централизация на разпръснатите публични данни (Раздел 2):**
Приоритет по стойност: НЗОК плащания, тръжна фаза ЦАИС ЕОП, декларации/финансиране на
Сметна палата, НСИ серии, НОИ пенсии.

**Фаза 4 — Отваряне на заключените регистри (Раздел 3):**
Публикуване като bulk open data вместо защита срещу масов достъп — с приоритет
законово публичните (декларации, изборни резултати, ГФО).

**Фаза 5 — Нови продукти (Раздел 4):**
Данни на ниво оферта; каноничен crosswalk на ключове; връзка възлагане→плащане.

**Хоризонтално изискване към целия портал:** всеки датасет да носи стабилен `updated_at`,
канонични ключове (ЕИК/ЕКАТТЕ), стабилен permalink и валидиран schema contract.
